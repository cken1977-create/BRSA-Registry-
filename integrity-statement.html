SHA-256 Measurement Integrity Statement

The cryptographic integrity model governing every readiness score produced by the Legacyline platform. Every assessment is sealed, verifiable, and tamper-evident.

01

Purpose

Every readiness score produced by the Legacyline platform is sealed with a SHA-256 cryptographic hash at the moment of computation. This hash is the integrity guarantee — it makes every assessment tamper-evident, reproducible, and independently verifiable.

This document defines the exact inputs to the hash function, the storage protocol, and the reproduction procedure. It is binding on all implementations of the BRSA readiness engine.

"A score without a hash is an opinion. A score with a hash is a fact."
02

Integrity Model

The integrity model operates on a simple principle — the hash is computed over a canonical string composed of all inputs and outputs of the scoring computation. If any value changes after the hash is computed, the hash will not match when verified. Tampering is immediately detectable.

The SHA-256 algorithm was selected for its collision resistance, widespread adoption, and suitability for audit environments. The hash is not encrypted — it is a one-way digest stored transparently alongside the score record.

03

Hash Input Specification

The canonical input string is constructed by concatenating the following fields in exactly this order, separated by pipe characters. All values are converted to strings before concatenation. Floating point scores are rounded to the nearest integer before inclusion.

Field 01
participant_id
Internal UUID of the participant record
Field 02
ruleset_version
e.g. "frari-individual-v1.0.0"
Field 03
composite_score
Integer 0–100
Field 04
evidence_score
Normalized sub-score · integer 0–100
Field 05
lifecycle_score
Normalized sub-score · integer 0–100
Field 06
intake_score
Normalized sub-score · integer 0–100
Field 07
consent_score
Discrete value · 0, 40, or 100
Field 08
computed_at
UTC timestamp · ISO 8601 format · e.g. "2026-04-25T14:32:00Z"
Canonical String Format
{participant_id}|{ruleset_version}|{composite_score}|{evidence_score}|{lifecycle_score}|{intake_score}|{consent_score}|{computed_at}
04

Storage Protocol

The SHA-256 digest is stored in the readiness_snapshots.score_hash column as a lowercase hexadecimal string. It is written in the same database transaction as the score record. The two are atomic — either both are written or neither is.

The hash column is write-once. No application path permits updating the hash after it is written. Any database-level modification would require direct access to the production database — which is logged, access-controlled, and audited separately.

05

Reproduction Procedure

Any assessment can be independently verified using the stored score record and the archived ruleset. The ReproduceAssessment() function in the Legacyline core engine performs this verification.

06

Integrity Guarantees

✓
Tamper Evidence
Any modification to a score record after sealing produces a hash mismatch that is immediately detectable during verification.
✓
Deterministic Reproduction
Given the same inputs in the same order, SHA-256 always produces the same output. Any party with access to the stored field values can independently verify the hash.
✓
Version Binding
The ruleset version is included in the hash input. A score computed under FRARI v1.0.0 cannot be falsely represented as computed under any other version.
✓
Timestamp Binding
The computed_at timestamp is included in the hash. Backdating an assessment record would produce a hash mismatch against the stored value.
✓
No Secret Keys
SHA-256 requires no secret key. The integrity guarantee does not depend on keeping any value confidential. Any party with the field values can verify the hash.
